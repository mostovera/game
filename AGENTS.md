# AGENTS.md — протокол координации агентов Sunnyside

> Репозиторий — одна игра **Sunnyside** (браузерная кооп-ферма-дайнер 50-х,
> R3F + Supabase) в корне. Этот файл — про две вещи: (1) как параллельные
> агенты договариваются друг с другом (канал координации, разделы 0–6) и
> (2) конвенции кода Sunnyside — владение файлами, границы модулей, тесты
> (разделы «Sunnyside» ниже). Читается **до** первой правки.
>
> Источник правил игры — `docs/specs/` (канон `00-canon.md` — истина при
> конфликте; клиент — `21-client.md`; бэкенд — `20-backend.md`).

Канал координации — таблицы `agents.memory` / `agents.tasks` / `agents.log` в
Supabase-проекте `pvautnecztynbnzrrdra`, через единственный вход:
**`scripts/agent-db.mjs`**. Напрямую SQL к этой схеме не ходим — только через
CLI, чтобы формат логов и задач был единым для всех агентов.

## 1. Перед началом работы (обязательно)

Первым действием в сессии — **всегда**:

```
node scripts/agent-db.mjs checkin --agent <имя>
```

(если работаете над конкретной задачей — `--task <id>` вторым аргументом).

Checkin печатает дайджест: открытые задачи, последние 20 записей лога, ключи
памяти. Прочитать его целиком **до** первой правки:

- **Открытые задачи** — посмотреть, что уже `in_progress` у других агентов.
  **Не брать чужую `in_progress`-задачу.** Если ваша работа собирается менять
  файлы, которые уже упомянуты в чужой активной задаче (поле `files` из её
  логов) — это **конфликт**, а не повод молча продолжать: заведите
  `log blocker` с описанием пересечения и решайте координацию (либо ждать,
  либо согласовать разделение зон) прежде чем писать код.
- **Последние логи** — контекст: что уже сделано, какие решения приняты, что
  сломано/заблокировано.
- **Ключи памяти** (`memory-list` / `memory-get`) — гочи, договорённости,
  эндпоинты, накопленные предыдущими агентами. Прочитать релевантные перед
  тем, как изобретать заново.

Затем — **зарегистрировать свою задачу**:

- Если задачи ещё нет — `task-add --title "..." --desc "..." [--owner <имя>]`.
- Если задача уже существует (кто-то завёл, вы её берёте) —
  `task-update --id <id> --status in_progress`.

Без этого шага работа не начинается — даже если задача кажется тривиальной.

## 2. Во время работы

- **`log decision`** — на каждое неочевидное решение: нейминг сущности,
  форма схемы (таблица/колонка/RPC), отклонение от спеки/канона, выбор между
  несколькими вариантами реализации. Пишите *почему*, не только *что*.
- **`log blocker`** — как только уткнулись: чужая задача блокирует файл,
  недостающий контракт, сломанный CI, неясность в спеке, конфликт с другим
  агентом. Не сидеть молча в ожидании — залогировать и, если применимо,
  обновить статус задачи на `blocked`.
- Långer-running работу стоит разбивать на `log start` в начале крупного
  этапа и промежуточные `log decision`/`log blocker` — так следующий агент
  (или человек), читающий лог, понимает ход мысли, а не только финал.

---

## 3. По завершении

Три обязательных шага, в этом порядке:

1. **`log finish --summary "..." --files a,b,c`** — что сделано и какие файлы
   затронуты (используется другими агентами для проверки пересечений на
   checkin).
2. **`task-update --id <id> --status done`** (или `review`, если нужна
   проверка человеком/другим агентом перед закрытием).
3. **`memory-set`** — для знаний, полезных *следующим* агентам: гочи, новые
   эндпоинты/RPC, договорённости о нейминге, открытые вопросы, которые вы
   разрешили. Если решение уже есть под существующим ключом — обновите его
   (`memory-set` с тем же `--key`), не плодите вариацию с новым именем.

Если задача не завершена, а вы прекращаете работу (конец сессии, передача
другому агенту) — `log handoff --summary "..."` вместо `finish`, с описанием
состояния и что осталось.

## 4. Гигиена памяти (`agents.memory`)

- **Ключ — `kebab-case`, специфичный и стабильный**: `supabase-rls-notes`,
  `truck-queue-endpoint`, не `notes` или `misc`.
- **Перед созданием ключа — `memory-get`/`memory-list`.** Если знание того же
  рода уже есть под другим ключом — обновите существующий, не создавайте
  дубль с похожим названием (`db-notes` и `database-notes` — не заводить оба).
- **Что хранить:** решения по схеме/API, известные гочи (что не работает
  очевидным способом и почему), договорённости о нейминге, ссылки на
  миграции/функции, открытые вопросы для следующего агента.
- **Что НЕ хранить — категорически:**
  - секреты, токены, ключи API, пароли, содержимое `.env*` — ни в `content`,
    ни в `title`, ни в `tags`. Токен Supabase достаётся только из Keychain
    (`security find-generic-password -s "Supabase Sunnyside PAT" -w`) и
    никогда не должен попасть в лог/память/коммит.
  - персональные данные пользователей.
  - большие вставки кода целиком (ссылка на файл + суть решения — достаточно).

## 5. Канал Supabase — только через CLI

Проект `pvautnecztynbnzrrdra`:

## 6. Примеры команд

```bash
# Начало сессии
node scripts/agent-db.mjs checkin --agent claude-sunnyside-ui

# Взять существующую задачу
node scripts/agent-db.mjs task-update --id 42 --status in_progress

# Завести новую задачу
node scripts/agent-db.mjs task-add \
  --title "RecipeBox: перенос в src/ui/kitchen" \
  --desc "Дубль src/ui/collections/RecipeBox.tsx устарел, канон — kitchen/" \
  --owner claude-sunnyside-ui

# Решение по ходу работы
node scripts/agent-db.mjs log --agent claude-sunnyside-ui --action decision \
  --summary "RecipeCard рендерит только kitchen-вариант; collections-дубль удалён" \
  --task 42 --files src/ui/kitchen/RecipeBox.tsx

# Блокер
node scripts/agent-db.mjs log --agent claude-sunnyside-ui --action blocker \
  --summary "Правка tokens.ts пересекается с задачей #37 (in_progress у net-агента)" \
  --task 42

# Завершение
node scripts/agent-db.mjs log --agent claude-sunnyside-ui --action finish \
  --summary "RecipeBox перенесён в kitchen/, дубль в collections/ удалён" \
  --task 42 --files src/ui/kitchen/RecipeBox.tsx,src/ui/collections/index.ts
node scripts/agent-db.mjs task-update --id 42 --status done

# Память для следующих агентов
node scripts/agent-db.mjs memory-set \
  --key recipebox-canon-location \
  --title "RecipeBox: канонический путь" \
  --content "Канон — src/ui/kitchen/RecipeBox.tsx. Версия в collections/ — удалена (была дублем до конвергенции), не восстанавливать." \
  --tags kitchen,collections,cleanup
```

# Sunnyside — конвенции кода

Специфика кодовой базы Sunnyside поверх протокола координации выше. Фундамент
(типы, контракты, сторы, каркас сцен) уже заложен архитектором — ты строишь **на нём**.

## S1. Золотые правила (нарушение = ревью-блок)

1. **Граница `логика ↔ рендер` жёсткая.** `types/`, `engine/`, `state/` не импортируют `three`/`@react-three/*` и не лезут в `scene/`. Проверяется `pnpm lint:boundary`. Нарушение — баг, не «прагматизм».
2. **Клиент не считает награду сам.** Любая мутация серверного стейта идёт через `BackendAdapter` (`engine/contracts.ts`). Оптимистичный патч — для мгновенности UI; истина — из ответа адаптера. Анти-чит (20-backend §3.7, 21-client §3.5).
3. **Время — только `serverNow()`** (`clock`-слайс), не `Date.now()` в игровой логике (21-client §3.6). Таймер «истёк» ≠ «начислено»: забор — отдельная подтверждаемая мутация.
4. **Балансы валют не персистятся** (анти-подмена, 21-client §3.4). Никогда не клади `econ.wallet` в persist.
5. **Не чини чужие файлы «мимоходом».** Нашёл проблему вне своей зоны владения (§S3) — заведи задачу/оставь `// TODO(owner):`, не редактируй трейлинг-правками. PR с диффом чужого модуля без согласования — отклоняется.
6. **Нейминг — из канона.** Любой новый ключ сущности вводится PR к `00-canon.md`, не выдумывается. В коде — только английский `snake_case` ключ; UI-строки — RU/EN через `ui.locale`.

## S2. Стек и команды

- **Стек (зафиксирован каноном D13/D14, 21-client §1):** Vite 5 · React 18 · `@react-three/fiber` 8 · `@react-three/drei` 9 · `three` 0.169 · `zustand` 5 · TypeScript strict · `@supabase/supabase-js` 2 · `idb` 8 · Tailwind 4 · vitest 2 · Playwright.
- **Пакетный менеджер — `pnpm`.** `package.json` в корне репозитория.
- **Не добавляй зависимости без спроса.** Список в `package.json` осознан; расширение — через согласование.

| Команда | Что делает |

## S3. Карта владения файлами (кто что пишет)

Каждый модуль — зона владения одного агента-волны. Внутрь чужой зоны не пишем; общаемся через **контракты** (`engine/contracts.ts`) и **типы** (`types/`).

| Зона | Путь | Владелец (роль) | Правила |
|---|---|---|---|
| **Домовые типы** | `src/types/**` | архитектура (стабильно) | словарь всей игры. Добавление типа — можно; ломающее переименование — согласование. Ноль импортов кроме внутри `types/`. |
| **Контракты систем** | `src/engine/contracts.ts` | архитектура (стабильно) | интерфейсы `*System` и `BackendAdapter`. Меняются редко и согласованно — от них зависят все. |
| **Чистые формулы** | `src/engine/econ/**` | эконом-агент | чистые функции (14-economy). **Обязательно** vitest ≥ 90% строк. Ноль сети/three. |
| **Системы** | `src/engine/<system>/**` | профильный агент по системе | реализуют интерфейс из `contracts.ts`. Мутации — только через `SystemContext.applyMutation`/`adapter`. |
| **Сторы** | `src/state/<slice>.ts` | агент слайса | скелеты заложены. Добавляй экшены/селекторы своего слайса. Не трогай чужой слайс. Персист — только через `partialize` в `state/index.ts` (согласование). |
| **Сетевой слой** | `src/net/**` | net-агенты (local/supabase) | `adapters/base.ts` — единый список методов; `local.ts`/`supabase.ts` переопределяют. Очередь (`queue.ts`), каналы (`channels.ts`), reconcile — сюда. |
| **Рендер** | `src/scene/<scene>/**` | scene-агент по сцене | R3F-графы. Читают стор селекторами, вызывают системы. Ноль бизнес-правил, ноль `@/net`. |
| **Ассеты-заглушки** | `src/assets/placeholders/**` | арт/сцен-агент | мастер-реестр `registry.ts` + `PlaceholderMesh.tsx` + `sound.ts`. См. §S6. `src/scene/assets/palette.ts` — отдельная общая палитра сцены (небо/свет/земля), не часть реестра ассетов. |
| **UI-оверлей** | `src/ui/**` | ui-агент | DOM поверх канваса (HUD, панели `ui_*`). Ноль three. `data-testid` для смоуков. |
| **Бутстрап** | `src/main.tsx`, `src/bootstrap/**` | net-bootstrap-агент | порядок инициализации (21-client §3.2). |
| **Композиция** | `src/App.tsx` | архитектура | свитч сцены + HUD. Правки согласуются. |
| **e2e** | `e2e/**` | qa-агент | Playwright-смоуки через `?screen=` + local-адаптер. |

**Общие файлы (`types/`, `engine/contracts.ts`, `state/index.ts`, `App.tsx`):** меняются только по согласованию — их правка задевает всех. Нужен новый метод адаптера или тип — предложи диффом, не втихую.

## S4. Правила импортов (граф зависимостей)

Разрешённое направление зависимостей (`A → B` = «A может импортировать B»):

```
scene/ ─┐
ui/    ─┼─→ engine/ ─→ types/
net/   ─┤     ↑
state/ ─┘     │
       └──────┘   (state → engine контракты, engine → types)
App.tsx, main.tsx — композиция, могут импортировать всё.
```

- `types/` — **лист графа**: ни от кого не зависит (кроме себя). Ноль `three`/react.
- `engine/` — зависит только от `types/`. Чистая логика/контракты, node-тестируема.
- `state/` — зависит от `types/` и `engine/` (контракты). **Не** импортирует `three`/`scene`/`ui`/`net`.
- `net/` — реализует `BackendAdapter` (`engine`), может писать в `state` (применять патчи). **Не** импортирует `scene`.
- `scene/` — рендер: читает `state` селекторами, вызывает системы (`engine`). **Не** импортирует `net` напрямую.
- `ui/` — DOM: читает `state`, вызывает системы. Ноль `three`, ноль `net`.

**Только через контракты.** Модуль не лезет во внутренности другого: сцена не импортирует внутренний хелпер сети; система не читает приватную переменную стора. Пересечение границы — через публичный экспорт (`@/types`, `@/engine`, `useStore`).

Алиас `@/` → `src/` (`import type { Plot } from '@/types'`).

## S5. Требования к тестам (vitest + Playwright)

**Уровень 1 — vitest (node, без браузера), обязателен:**
- `engine/econ/**` — **≥ 90% покрытия строк** (гейт в `vite.config.ts`). Это деньги игроков — ошибка недопустима. Пример: `engine/econ/econ.test.ts`.
- `state/<slice>` — редьюсеры как чистые функции: оптимистичная дельта + inverse-откат, дренаж очереди (мок). Пример: `state/store.test.ts`.
- `engine/clock`, `bootstrap/debug` и прочая чистая логика — покрыты на границах (перевод часов не ускоряет; невалидный параметр отбрасывается). Пример: `bootstrap/debug.test.ts`.
- Файлы тестов: `*.test.ts` рядом с кодом. Окружение `node` (граница гарантирует запуск без three).

**Уровень 2 — Playwright смоуки (`e2e/**`):** каждый ключевой экран открывается через `?screen=` + local-адаптер, монтируется (`canvas` есть), нет `console-error`, ключевой `data-testid` присутствует, базовый клик даёт отклик. **Не** пиксельно (это дизайн-ревью). Пример: `e2e/smoke.spec.ts`.

**Правило:** новый модуль `engine/`/`state/` без теста — незавершённая работа. `scene/`/`ui/` — покрываются смоуком соответствующего экрана.

## S6. Политика заглушек-ассетов

До прихода финальных GLB (Blender-пайплайн, 22-audio-visual) **всё рисуется примитивами**.

- **Реестр (единственный, с registry-converge) — `src/assets/placeholders/registry.ts`:** `id → AssetEntry { category, placeholder?, final: FinalAssetRequirements, ... }`. Ключи = ключи канона (`bld_*`, `tp_*`, `an_*`, `mch_*`, `toy_*`, `cos_*`, `st_*`, `npc_*`, `staff_*`) плюс рабочие категории вне канон-реестра (`crop_*`, `env_*`, `vfx_*`, `ui_*`…). `tp_*` project-объекты — по стадиям стройки, ключ `${tp_key}_stage${1|2|3}` (леса/каркас/готово), не общий стенд-ин.
- **Рендер — `src/assets/placeholders/PlaceholderMesh.tsx`:** `<PlaceholderMesh id="bld_diner" />`. Компонент **не знает** конкретный ассет — только id. Все 4 сцены (`farm`/`town`/`fair`/`shift`) рисуют через него.
- **`final` в записи — пока ТРЕБОВАНИЯ к финалу** (стиль/tris/формат, Фаза D), не путь к GLB.
- **Цвета — только именованные из `CANON_PALETTE`** (`color('pal_cherry')`), не hex в компонентах. Эстетика: `MeshLambertMaterial`, плоские цвета, ноль PBR.
- **`public/placeholders/`** пуст по замыслу — заглушки процедурны (см. его README).
- Нет ассета в реестре → `PlaceholderMesh` рисует magenta-маркер + dev-лейбл (заметно в dev), не падает (C8/V3).
- Таблица всех ассетов — автоген `scripts/gen-assets-table.mjs` → `docs/ASSETS.md` (не редактировать руками).

Добавляешь сущность канона — заведи её запись в `src/assets/placeholders/registry.ts` (хотя бы placeholder) и синхронизируй с таблицей ассетов спеки 22-audio-visual.

## S7. Перф-бюджеты (21-client §3.9)

Цель — 60 fps на среднем ноутбуке 2020 (Intel Iris/M1, 1080p); `town` — ≥ 40 fps. На кадр (мастер-числа — 21-client §3.9): draw calls ≤ 120 (farm)/150 (town); треугольники ≤ 250k/350k; уникальные материалы ≤ 10–12; текстуры 0.

Обязательные техники: **инстансинг** повторяющихся пропсов (грядки, деревья, гости, фермы города); один `AnimationMixer` на тип; город — LOD/имперостеры; code-split по сценам (`React.lazy`), выгрузка GLB при уходе; `<PerfGate>` → авто-`liteMode` при < 45 fps 2 с; адаптивный `dpr [1, 1.5]`. Профилировка — `?perf=hud` (r3f-perf).

## S8. Сеть, оптимистичность, оффлайн (21-client §3.5)

- Мутация: `optimistic → queued → in-flight → confirmed` (happy path); `→ conflict → rollback + мягкий тост` (никогда не красный, P3); `→ offline → ждём online`.
- Очередь **персистится в IndexedDB** (`net/queue.ts`, `idb`), дренажится **FIFO** (причинность). Переживает перезагрузку/закрытие вкладки.
- `clientMutationId` — локальный ключ элемента очереди. Серверный дедуп — state-guards RPC + `currency_ledgers.idempotency_key` (20-backend §3.4).
- Reconnect: экспоненциальный backoff (`NET_TIMINGS.backoff`), пересоздание Realtime, дельта-ресинк town/fair/event, повторный замер `serverOffset`.
- Дебаг-параметры (`?screen=`, `?net=offline`, `?seed=`, `?clock=` …) — только dev/e2e (`isDebugEnabled()`), игнор в проде.

## S9. Разное

- **Локализация:** строки UI — RU/EN через `ui.locale` (дефолт `ru`, канон §5). Ключи канона в коде — английские.
- **Тон (канон §1.2, пилларсы P1–P4):** дружелюбно, без наказаний, «ничего не сгорает». Тексты ошибок — тёплые.
- **Env:** ключи из `import.meta.env` (`.env.sunnyside`, в `.gitignore`; шаблон — `.env.sunnyside.example`). Секретный ключ Supabase в клиент **не** кладём — только `VITE_SUPABASE_PUBLISHABLE_KEY`.
