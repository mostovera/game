# AGENTS.md — конвенции код-агентов Sunnyside

> Этот файл — договор для параллельных агентов, пишущих код в `sunnyside/`.
> Читается **до** первой правки. Источник правил игры — `docs/specs/` (канон
> `00-canon.md` — истина при конфликте; клиент — `21-client.md`; бэкенд — `20-backend.md`).
> Фундамент (типы, контракты, сторы, каркас сцен) уже заложен архитектором — ты строишь **на нём**.

---

## 0. Золотые правила (нарушение = ревью-блок)

1. **Прототип в корне репозитория (`../src`, `farm-truck`) — НЕ ТРОГАТЬ ВООБЩЕ.** Мы живём только в `sunnyside/`.
2. **Граница `логика ↔ рендер` жёсткая.** `types/`, `engine/`, `state/` не импортируют `three`/`@react-three/*` и не лезут в `scene/`. Проверяется `pnpm lint:boundary`. Нарушение — баг, не «прагматизм».
3. **Клиент не считает награду сам.** Любая мутация серверного стейта идёт через `BackendAdapter` (`engine/contracts.ts`). Оптимистичный патч — для мгновенности UI; истина — из ответа адаптера. Анти-чит (20-backend §3.7, 21-client §3.5).
4. **Время — только `serverNow()`** (`clock`-слайс), не `Date.now()` в игровой логике (21-client §3.6). Таймер «истёк» ≠ «начислено»: забор — отдельная подтверждаемая мутация.
5. **Балансы валют не персистятся** (анти-подмена, 21-client §3.4). Никогда не клади `econ.wallet` в persist.
6. **Не чини чужие файлы «мимоходом».** Нашёл проблему вне своей зоны владения (§2) — заведи задачу/оставь `// TODO(owner):`, не редактируй трейлинг-правками. PR с диффом чужого модуля без согласования — отклоняется.
7. **Нейминг — из канона.** Любой новый ключ сущности вводится PR к `00-canon.md`, не выдумывается. В коде — только английский `snake_case` ключ; UI-строки — RU/EN через `ui.locale`.

- **Стек (зафиксирован каноном D13/D14, 21-client §1):** Vite 5 · React 18 · `@react-three/fiber` 8 · `@react-three/drei` 9 · `three` 0.169 · `zustand` 5 · TypeScript strict · `@supabase/supabase-js` 2 · `idb` 8 · Tailwind 4 · vitest 2 · Playwright.

## 2. Карта владения файлами (кто что пишет)

Каждый модуль — зона владения одного агента-волны. Внутрь чужой зоны не пишем; общаемся через **контракты** (`engine/contracts.ts`) и **типы** (`types/`).

| Зона | Путь | Владелец (роль) | Правила |
|---|---|---|---|
| **Домовые типы** | `src/types/**` | архитектура (стабильно) | словарь всей игры. Добавление типа — можно; ломающее переименование — согласование. Ноль импортов кроме внутри `types/`. |
| **Контракты систем** | `src/engine/contracts.ts` | архитектура (стабильно) | интерфейсы `*System` и `BackendAdapter`. Меняются редко и согласованно — от них зависят все. |
| **Чистые формулы** | `src/engine/econ/**` | эконом-агент | чистые функции (14-economy). **Обязательно** vitest ≥ 90% строк. Ноль сети/three. |
| **Системы** | `src/engine/<system>/**` | профильный агент по системе | реализуют интерфейс из `contracts.ts`. Мутации — только через `SystemContext.applyMutation`/`adapter`. |
| **Сторы** | `src/state/<slice>.ts` | агент слайса | скелеты заложены. Добавляй экшены/селекторы своего слайса. Не трогай чужой слайс. Персист — только через `partialize` в `state/index.ts` (согласование). |
| **Сетевой слой** | `src/net/**` | net-агенты (local/supabase) | `adapters/base.ts` — единый список методов; `local.ts`/`supabase.ts` переопределяют. Очередь (`queue.ts`), каналы (`channels.ts`), reconcile — сюда. |
| **Рендер** | `src/scene/<scene>/**` | scene-агент по сцене | R3F-графы. Читают стор селекторами, вызывают системы. Ноль бизнес-правил, ноль `@/net`. |
| **Ассеты-заглушки** | `src/scene/assets/**` | арт/сцен-агент | реестр `registry.ts` + `Prop.tsx` + `palette.ts`. См. §5. |
| **UI-оверлей** | `src/ui/**` | ui-агент | DOM поверх канваса (HUD, панели `ui_*`). Ноль three. `data-testid` для смоуков. |
| **Бутстрап** | `src/main.tsx`, `src/bootstrap/**` | net-bootstrap-агент | порядок инициализации (21-client §3.2). |
| **Композиция** | `src/App.tsx` | архитектура | свитч сцены + HUD. Правки согласуются. |
| **e2e** | `e2e/**` | qa-агент | Playwright-смоуки через `?screen=` + local-адаптер. |

**Общие файлы (`types/`, `engine/contracts.ts`, `state/index.ts`, `App.tsx`):** меняются только по согласованию — их правка задевает всех. Нужен новый метод адаптера или тип — предложи диффом, не втихую.

## 3. Правила импортов (граф зависимостей)

Разрешённое направление зависимостей (`A → B` = «A может импортировать B»):

```
scene/ ─┐
ui/    ─┼─→ engine/ ─→ types/
net/   ─┤     ↑
state/ ─┘     │
       └──────┘   (state → engine контракты, engine → types)
App.tsx, main.tsx — композиция, могут импортировать всё.
```

- `types/` — **лист графа**: ни от кого не зависит (кроме себя). Ноль `three`/react.
- `engine/` — зависит только от `types/`. Чистая логика/контракты, node-тестируема.
- `state/` — зависит от `types/` и `engine/` (контракты). **Не** импортирует `three`/`scene`/`ui`/`net`.
- `net/` — реализует `BackendAdapter` (`engine`), может писать в `state` (применять патчи). **Не** импортирует `scene`.
- `scene/` — рендер: читает `state` селекторами, вызывает системы (`engine`). **Не** импортирует `net` напрямую.
- `ui/` — DOM: читает `state`, вызывает системы. Ноль `three`, ноль `net`.

**Только через контракты.** Модуль не лезет во внутренности другого: сцена не импортирует внутренний хелпер сети; система не читает приватную переменную стора. Пересечение границы — через публичный экспорт (`@/types`, `@/engine`, `useStore`).

Алиас `@/` → `src/` (`import type { Plot } from '@/types'`).

## 4. Требования к тестам (vitest + Playwright)

**Уровень 1 — vitest (node, без браузера), обязателен:**
- `engine/econ/**` — **≥ 90% покрытия строк** (гейт в `vite.config.ts`). Это деньги игроков — ошибка недопустима. Пример: `engine/econ/econ.test.ts`.
- `state/<slice>` — редьюсеры как чистые функции: оптимистичная дельта + inverse-откат, дренаж очереди (мок). Пример: `state/store.test.ts`.
- `engine/clock`, `bootstrap/debug` и прочая чистая логика — покрыты на границах (перевод часов не ускоряет; невалидный параметр отбрасывается). Пример: `bootstrap/debug.test.ts`.
- Файлы тестов: `*.test.ts` рядом с кодом. Окружение `node` (граница гарантирует запуск без three).

**Уровень 2 — Playwright смоуки (`e2e/**`):** каждый ключевой экран открывается через `?screen=` + local-адаптер, монтируется (`canvas` есть), нет `console-error`, ключевой `data-testid` присутствует, базовый клик даёт отклик. **Не** пиксельно (это дизайн-ревью). Пример: `e2e/smoke.spec.ts`.

**Правило:** новый модуль `engine/`/`state/` без теста — незавершённая работа. `scene/`/`ui/` — покрываются смоуком соответствующего экрана.

## 5. Политика заглушек-ассетов

До прихода финальных GLB (Blender-пайплайн, 22-audio-visual) **всё рисуется примитивами**.

- **Реестр — `src/scene/assets/registry.ts`:** `assetKey → { placeholder, final? }`. Ключи = ключи канона (`bld_*`, `tp_*`, `toy_*`, `cos_*`, `st_*`, `env_*`).
- **Рендер — `src/scene/assets/Prop.tsx`:** `<Prop assetKey="bld_diner" />`. Компонент **не знает** конкретный ассет — только ключ.
- **Замена заглушки на финал = добавить путь в поле `final` ОДНОЙ строки** реестра. GLB кладётся в `public/assets/props/`. Ноль правок в сценах.
- **Цвета — только именованные из `public/palette.json`** (`color('diner_red')`), не hex в компонентах. Эстетика: `MeshLambertMaterial`, плоские цвета, ноль PBR (наследие прототипа).
- **`public/placeholders/`** пуст по замыслу — заглушки процедурны (см. его README).
- Нет ассета в реестре → `Prop` рисует magenta-маркер (заметно в dev), не падает (C8).

Добавляешь сущность канона — заведи её запись в `registry.ts` (хотя бы placeholder) и синхронизируй с реестром `placeholder→final` спеки 22-audio-visual.

## 6. Перф-бюджеты (21-client §3.9)

Цель — 60 fps на среднем ноутбуке 2020 (Intel Iris/M1, 1080p); `town` — ≥ 40 fps. На кадр (мастер-числа — 21-client §3.9): draw calls ≤ 120 (farm)/150 (town); треугольники ≤ 250k/350k; уникальные материалы ≤ 10–12; текстуры 0.

Обязательные техники: **инстансинг** повторяющихся пропсов (грядки, деревья, гости, фермы города); один `AnimationMixer` на тип; город — LOD/имперостеры; code-split по сценам (`React.lazy`), выгрузка GLB при уходе; `<PerfGate>` → авто-`liteMode` при < 45 fps 2 с; адаптивный `dpr [1, 1.5]`. Профилировка — `?perf=hud` (r3f-perf).

## 7. Сеть, оптимистичность, оффлайн (21-client §3.5)

- Мутация: `optimistic → queued → in-flight → confirmed` (happy path); `→ conflict → rollback + мягкий тост` (никогда не красный, P3); `→ offline → ждём online`.
- Очередь **персистится в IndexedDB** (`net/queue.ts`, `idb`), дренажится **FIFO** (причинность). Переживает перезагрузку/закрытие вкладки.
- `clientMutationId` — локальный ключ элемента очереди. Серверный дедуп — state-guards RPC + `currency_ledgers.idempotency_key` (20-backend §3.4). Универсальный `client_mutation_id` в бэкенде — открытый вопрос (21-client §8 п.1).
- Reconnect: экспоненциальный backoff (`NET_TIMINGS.backoff`), пересоздание Realtime, дельта-ресинк town/fair/event, повторный замер `serverOffset`.
- Дебаг-параметры (`?screen=`, `?net=offline`, `?seed=`, `?clock=` …) — только dev/e2e (`isDebugEnabled()`), игнор в проде.

## 8. Разное

- **Локализация:** строки UI — RU/EN через `ui.locale` (дефолт `ru`, канон §5). Ключи канона в коде — английские.
- **Тон (канон §1.2, пилларсы P1–P4):** дружелюбно, без наказаний, «ничего не сгорает». Тексты ошибок — тёплые.
- **Env:** ключи из `import.meta.env` (`.env.sunnyside`, в `.gitignore`). Секретный ключ Supabase в клиент **не** кладём — только `VITE_SUPABASE_PUBLISHABLE_KEY`.
- **Git не трогаем** — коммитит оркестратор. Не запускай `pnpm install` вне смоук-фазы.
- **Спорные развилки** — в `docs/specs/OPEN-QUESTIONS.md` и разделах «Открытые вопросы» спек. Не переоткрывай закрытое каноном (`00-canon.md §4`).
